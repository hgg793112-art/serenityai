# 類 Tolan 對話記憶 + 溝通 — 免費方案

目標：在**不付費**前提下，實現「有記憶的陪伴對話」（文字為主，可選語音），架構參考 Tolan，全部使用免費額度或開源組件。

---

## 一、推薦方案總覽（全免費）

| 層級 | 用途 | 選型 | 說明 |
|------|------|------|------|
| **對話生成** | 小寧回覆內容 | **Gemini 免費額度**（或 Groq 免費） | 專案已接 Gemini，免費額度足夠個人/小規模使用。 |
| **對話記憶** | 近期對話 + 長期記憶 | **近期**：資料表存最近 N 輪<br>**長期**：可選向量檢索 | 先做「近期 N 輪」即可上線；要更像 Tolan 再加向量記憶。 |
| **記憶儲存** | 存對話與記憶片段 | **Supabase 免費版** | 已有 Supabase，免費額度內新增表即可。 |
| **向量檢索（可選）** | 長期記憶檢索 | **Supabase pgvector + Gemini 免費 embedding** | 用 `text-embedding-004` / 免費 embedding 模型寫入向量，查詢時按相似度取回。 |

**結論**：不付費可以實現。優先做「近期對話 + 簡單長期事實」，再視需要加「向量長期記憶」。

---

## 二、架構說明

### 2.1 對話流程（每輪）

1. 使用者發送一則訊息。
2. **組裝上下文**：
   - **近期**：從 Supabase 讀取該使用者最近 N 條對話（例如 20 條）。
   - **長期（若已做）**：用當前訊息做 embedding，在 pgvector 查詢 top-k 條相關記憶，或從「事實表」取幾條摘要。
3. **系統提示**：固定人設（小寧／刺蝟 IP）+ 可選的「已知關於用戶的事」。
4. 呼叫 **Gemini（或 Groq）** 生成回覆。
5. **寫回**：把本輪 user / assistant 寫入對話表；可選：用 LLM 抽一兩句「可長期記住的事」寫入記憶表或向量表。

### 2.2 記憶分兩層（與 Tolan 對齊概念）

- **短期／近期**：最近 N 輪對話原文（或摘要），直接塞進 prompt。  
  - 存：`chat_messages` 表。  
  - 不付費、不佔向量額度。
- **長期**：  
  - **簡版**：一個 `memory_facts` 表，存「關於用戶的一句話事實」，每輪可檢索最近幾條或全部（條數少時）。  
  - **進階**：用 **Gemini 免費 embedding** 寫入 **Supabase pgvector**，每輪用當前句查相似記憶，再塞進 prompt。  

先做短期 + 簡單長期即可「有記憶的對話」；要更像 Tolan 再上向量。

---

## 三、資料表設計（Supabase）

在現有專案下新增（不影響既有 `mood_logs`）：

- **chat_messages**：每輪對話  
  - `id`, `user_id`（或 device_id）, `role`（user/assistant）, `content`, `created_at`  
  - 用於「近期 N 條」。
- **memory_facts**（簡單長期）：  
  - `id`, `user_id`, `fact_text`, `source_message_id`（可選）, `created_at`  
  - 可選：後台用 LLM 從對話中抽取「要記住的事」寫入。
- **可選 - 向量長期記憶**：  
  - 啟用 Supabase **pgvector**，新增表例如 `memory_embeddings (user_id, content, embedding vector(768), created_at)`。  
  - 寫入：用 Gemini 免費 embedding 算向量後 insert。  
  - 查詢：`ORDER BY embedding <-> query_embedding LIMIT k`。

使用者識別：若無登入，可用 **localStorage 的 device_id**（或匿名 id）當 `user_id`，與現有心情日誌邏輯一致。

---

## 四、免費額度與限制

- **Gemini**：免費額度有 RPM/日限，足夠個人或小規模；語音 Live 若用同一 key 也佔額度。  
- **Supabase**：免費版有 DB 大小與請求數限制，對話 + 少量記憶表完全夠用。  
- **Groq**：若選用，多數情況為免費額度，可作為 Gemini 的備援或替代。  

不付費條件下，只要控制「每用戶每日對話量」與「記憶條數」，即可長期使用。

## 五、實作順序建議

1. **Phase 1（最小可行）**  
   - 新增 `chat_messages` 表與前端「小寧聊天」對話 UI（輸入框 + 氣泡）。  
   - 每輪：讀最近 20 條 → 組 prompt → 呼叫 Gemini（或 Groq）→ 寫回兩條（user/assistant）。  
   - 不付費：僅用 Gemini 免費額度 + Supabase 免費版。

2. **Phase 2（簡單長期記憶）**  
   - 新增 `memory_facts` 表。  
   - 每輪結束後可選：用 LLM 從本輪對話抽 0～1 條「要記住的事」寫入。  
   - 下次組 prompt 時帶上最近 5～10 條 facts。  
   - 仍無需向量、無需付費。

3. **Phase 3（可選，更像 Tolan）**  
   - 啟用 pgvector，新增 `memory_embeddings`。  
   - 用 Gemini 免費 embedding 寫入重要記憶；每輪用當前句查 top-k 條相似記憶放進 prompt。  
   - 仍全部在免費額度內。

## 六、小結

- **不付費可以實現**類 Tolan 的「對話記憶 + 溝通」：用 **Gemini（或 Groq）免費 + Supabase 免費**，先做近期對話 + 簡單事實記憶，再可選加向量長期記憶。  
- 語音部分：可維持現有 **Gemini Live**（同一 key 免費額度內），或改為「語音→文字→Gemini→文字→語音」以省額度。  
- 實作上優先完成 **Phase 1**，再依需要加 Phase 2 / 3。
